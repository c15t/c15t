# React v3 Benchmark Notes

This document tracks the benchmark evidence for the `@c15t/react` v3 split API work.

The main comparison is:

- v2: common public import path from `@c15t/react`, including provider and UI components.
- v3: split import paths from `@c15t/react/v3`, loading only the provider, hooks, UI, and module hooks needed by the scenario.

## How To Run

Run the important React v2/v3 browser benchmarks together:

```bash
bun run bench:important-react -- --iterations 50
```

Run with explicit warmup and script-count cases:

```bash
bun run bench:important-react -- --iterations 50 --warmup 3 --script-counts 5,10,25,50
```

This runs two suites in parallel:

- `react-browser-bench`: banner readiness and visibility timing.
- `script-lifecycle-bench`: script loading timing after accepting consent.

JSON output is written to:

- `.benchmarks/current/banner-visibility/react-v2-v3-banner-visibility.json`
- `.benchmarks/current/script-count/react-v2-v3-script-count.json`

Bundle size is measured separately by `benchmarks/bundle-test-app`, because it is build analysis rather than an iteration-based browser runtime benchmark.

## Reference Run

Reference numbers below are from a local 50-iteration run with 3 warmup iterations on April 28, 2026 (round 2).

These browser benchmarks use deterministic local routes and preseeded state. They are intended to compare c15t React runtime overhead, not third-party CDN latency.

## Banner Visibility

Scenario:

- v2 renders `ConsentManagerProvider` and `ConsentBanner` from `@c15t/react`.
- v3 renders the provider, hooks, and banner from split `@c15t/react/v3` entrypoints.
- Backend/network state is preseeded so the result focuses on render, hydration, and visible banner readiness.

| Metric | v2 median | v3 median | Change | v2 p95 | v3 p95 | Change |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| `bannerReadyMs` | 50.6 ms | 41.0 ms | -19.0% | 54.6 ms | 48.0 ms | -12.1% |
| `bannerVisibleMs` | 50.6 ms | 41.0 ms | -19.0% | 54.6 ms | 48.0 ms | -12.1% |
| `mountMs` | 39.4 ms | 33.9 ms | -14.0% | 43.1 ms | 39.6 ms | -8.1% |

Takeaway: v3 holds its absolute banner-visible median at ~41 ms across both round-1 and round-2 passes. The headline percentage moved because v2 ran faster on this machine that day (cross-run JIT/warmup variance, not a v2 improvement). The wins from this surface that matter live in code: the 10 ms `setTimeout` floor and the SSR-mount paint cycle that round 1 removed, plus the `useLayoutEffect` mount gate now applied to `ConsentDialog`, `IABConsentBanner`, `IABConsentDialog`, and `Frame` (round 2). The Frame component additionally lost a `requestAnimationFrame`-gated render that was costing ~16 ms on every mount; that win doesn't appear in this benchmark because no Frame is rendered.

## Script Loading

Scenario:

- v2 uses `@c15t/react` with `ConsentManagerProvider` and `store.scripts`.
- v3 uses `@c15t/react/v3/provider`, `@c15t/react/v3/hooks`, and `@c15t/react/v3/module-hooks/script-loader`.
- The benchmark clicks "Accept all" and measures in-page time until every configured local script has executed.
- Loaded-script IDs and DOM IDs are collected after the timed completion marker, so the measured path does not include frame polling, repeated DOM queries, or benchmark-state JSON rendering.

| Scripts | v2 median | v3 median | Change | v2 p95 | v3 p95 | Change |
| ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| 5 | 12.8 ms | 10.0 ms | -21.9% | 16.0 ms | 11.5 ms | -28.1% |
| 10 | 13.4 ms | 10.5 ms | -21.6% | 15.5 ms | 12.1 ms | -21.9% |
| 25 | 19.4 ms | 12.4 ms | -36.1% | 22.8 ms | 19.2 ms | -15.8% |
| 50 | 26.8 ms | 18.8 ms | -29.9% | 30.7 ms | 21.1 ms | -31.3% |

Takeaway: v3 stays faster than v2 across script counts. Medians are within run-to-run noise of round 1; the `DocumentFragment` batching helps real pages with many simultaneous category grants but doesn't move this benchmark, where wall time is dominated by browser fetch + parse + execute.

## Off-bench wins (round 2)

Several round-2 fixes target surfaces this benchmark does not exercise. Their wins are real but require the corresponding scenario to surface:

- **`IABConsentBanner`**: same setTimeout/mount-gate fix as the regular banner — same ~30 % bannerVisibleMs improvement, but only for IAB-mode users.
- **`ConsentDialog` / `IABConsentDialog` mount gate**: saves a paint cycle on every dialog open.
- **`Frame`**: removed the `isReady` + `requestAnimationFrame` gate. Saves ~16 ms on every Frame mount.
- **Persistence sync hydrate**: `usePersistence` now calls `handle.hydrate()` inside the lazy `useState` initializer instead of a `useEffect`. Returning visitors no longer see a flash of default consent state before stored consents apply.
- **GVL processing cache**: the ~10K-iteration GVL → UI mapping inside `IABConsentDialog` is hoisted to a module-level `WeakMap` cache keyed on the kernel GVL reference, so reopening the dialog (or remounting it) reuses the prior result.
- **Provider theme `deepMerge` stabilization**: the merge of user theme onto `defaultTheme` is now memoized on `options.theme` directly rather than the full `options` object, so callers passing fresh `options` props don't pay for a deepMerge per render.
- **Kernel `set.language` no-op bail**: skips the snapshot allocation and subscriber notification when the language is set to its current value.

## Deferred items

These were flagged in the perf audit but are larger refactors and were not landed in round 2:

- **#8 Dynamic IAB import**: `@c15t/iab/v3` is currently imported unconditionally by `iab-context.tsx`, so its ~15 KB ships with every v3 bundle even when IAB isn't configured. Needs a lazy/dynamic-import boundary.
- **#9 Narrow selectors in `useConsentManager`**: the hook reads the full kernel snapshot today; switching consumers to slice selectors would reduce re-renders. Would need an audit of every caller.
- **#11 Per-locale translation splitting**: the `@c15t/translations` package imports all 34 locale files synchronously. Per-locale code-splitting helps large multilingual sites; needs work in the translations package.

## Bundle Context

The bundle benchmark answers a different question: how much client JavaScript Next includes for each import pattern.

Measured route-level client gzip additions:

| Scenario | Total client JS gzip | c15t addition |
| --- | ---: | ---: |
| v2 standard root `@c15t/react` full route | 236,441 B | 47,982 B |
| v3 split route + default persistence | 240,343 B | 51,884 B |
| v3 standard + persistence + script loader route | 244,250 B | 55,791 B |
| v3 full harness route | 245,907 B | 57,448 B |

This should be read carefully:

- These are client bundle numbers, not total page weight and not runtime script execution speed.
- The v3 split API is designed so apps can import only the v3 pieces they use.
- A route that intentionally imports more v3 modules can still be larger than the v2 baseline.
- The common v3 path should be measured as provider + UI + script loader, without optional network blocker, iframe blocker, or IAB modules unless the app uses them.

## Current Interpretation

The v3 work is showing the intended runtime improvement:

- Banner visibility is faster.
- Script loading after consent is faster.
- The split API gives us a cleaner way to keep optional capabilities out of common app paths.

The bundle-size story is more nuanced:

- v3 is not automatically smaller in every synthetic route.
- Import shape matters more in v3 because the API is intentionally split.
- The benchmark suite should continue to track common paths separately from full-harness paths.
