<section id="overview">
  Different countries need different consent experiences. Policy packs let you define those rules once — c15t picks the right one automatically based on where the visitor is.

  A policy pack is an ordered array of `PolicyConfig` objects. Each policy targets a region, country, or jurisdiction and controls:

  - the consent model (`opt-in`, `opt-out`, `none`, or `iab`)
  - which consent categories are in scope
  - which banner and dialog actions are shown
  - which translation profile is used
  - which proof fields are stored for the consent record

  <Callout type="info">
    Backend policy packs use `policyPacks`. Frontend offline previews use `offlinePolicy.policies`. Both accept the same `PolicyConfig` shape.
  </Callout>
</section>

<section id="quickstart">
  ## Quickstart

  The fastest way to get started is with the built-in defaults:

  ```ts
  import { policyPackPresets } from 'c15t';

  const policies = [
    policyPackPresets.europeOptIn(),    // GDPR opt-in banner
    policyPackPresets.californiaOptOut(), // CCPA opt-out banner
    policyPackPresets.worldNoBanner(),          // No banner elsewhere
  ];
  ```

  | Preset | Model | UI | Matches |
  |--------|-------|----|---------|
  | `europeOptIn()` | `opt-in` | banner | EU/EEA jurisdictions |
  | `europeIab()` | `iab` | banner | EU/EEA (TCF 2.3) |
  | `californiaOptOut()` | `opt-out` | banner | US-CA region |
  | `quebecOptIn()` | `opt-in` | banner | CA-QC region |
  | `worldNoBanner()` | `none` | none | default fallback |

  These are starting points. Customize them or write your own `PolicyConfig` objects for production.
</section>

<section id="what-users-see">
  ## What Users See

  Each policy combination produces a different consent experience:

  | Policy Config | User Experience |
  |--------------|-----------------|
  | `model: 'opt-in'`, `ui.mode: 'banner'` | Banner appears, nothing loads until the user consents |
  | `model: 'opt-out'`, `ui.mode: 'banner'` | Banner appears, everything loads immediately — user can opt out |
  | `model: 'none'`, `ui.mode: 'none'` | No banner, all categories auto-granted silently |
  | `model: 'opt-in'`, `ui.mode: 'dialog'` | Full-screen dialog, nothing loads until the user consents |
  | `model: 'iab'`, `ui.mode: 'banner'` | IAB TCF 2.3 banner with vendor-level controls |
</section>

<section id="resolution-flow">
  ## How Policy Resolution Works

  <Tabs items={['Simplified', 'Sequence Diagram']}>
    <Tab value="Simplified">
      1. **Visitor arrives** — c15t reads the visitor's country, region, and jurisdiction from request headers or geo overrides
      2. **Match by region** — checks for a policy targeting the specific region (e.g., US-CA, CA-QC)
      3. **Match by country** — if no region match, checks for a country-level policy (e.g., US, DE)
      4. **Match by jurisdiction** — if no country match, checks jurisdiction codes (e.g., GDPR, CCPA)
      5. **Fall back to default** — if nothing matches, uses the policy marked `isDefault: true`
      6. **No match, no default** — resolves to no-banner mode (silent, no consent UI)
    </Tab>

    <Tab value="Sequence Diagram">
      ```mermaid
      sequenceDiagram
          participant Visitor as Visitor
          participant C15t as c15t Runtime
          participant Pack as Policy Pack
          participant UI as Consent UI

          Visitor->>C15t: Request (country=US, region=CA)
          C15t->>Pack: Check region matchers
          alt Region match found
              Pack-->>C15t: policy_ca_opt_out
          else No region match
              C15t->>Pack: Check country matchers
              alt Country match found
                  Pack-->>C15t: policy_us_banner
              else No country match
                  C15t->>Pack: Check jurisdiction matchers
                  alt Jurisdiction match found
                      Pack-->>C15t: policy_gdpr_opt_in
                  else No jurisdiction match
                      C15t->>Pack: Check default policy
                      alt Default exists
                          Pack-->>C15t: policy_default_world
                      else No default
                          Pack-->>C15t: No-banner mode
                      end
                  end
              end
          end

          C15t->>C15t: Compute fingerprint
          C15t->>UI: Render resolved policy
      ```
    </Tab>
  </Tabs>

  Within the same matcher type, the first policy in the array wins. Pack order matters when two policies target the same country or region.

  <Callout type="warn">
    Only one default policy is allowed. Use `inspectPolicies()` to surface overlapping matchers and other warnings before deployment.
  </Callout>
</section>

<section id="anatomy">
  ## Anatomy of a Policy

  Each `PolicyConfig` has five parts:

  ```ts
  import type { PolicyConfig } from '@c15t/schema/types';

  const euPolicy: PolicyConfig = {
    // 1. match — where this policy applies
    id: 'policy_eu_opt_in',
    match: {
      jurisdictions: ['GDPR'],
    },

    // 2. consent — what consent model and categories to enforce
    consent: {
      model: 'opt-in',
      expiryDays: 365,
      scopeMode: 'strict',
      categories: ['necessary', 'measurement', 'marketing'],
      preselectedCategories: ['necessary'],
    },

    // 3. ui — how the banner and dialog behave
    ui: {
      mode: 'banner',
      banner: {
        allowedActions: ['accept', 'reject', 'customize'],
        primaryAction: 'customize',
        uiProfile: 'compact',
      },
      dialog: {
        allowedActions: ['accept', 'reject', 'customize'],
      },
    },

    // 4. i18n — which language/profile to use for copy
    i18n: {
      messageProfile: 'eu',
    },

    // 5. proof — which signals to store alongside consent evidence
    proof: {
      storeIp: true,
      storeUserAgent: true,
      storeLanguage: true,
    },
  };
  ```

  All parts except `id` and `match` are optional. Omitted sections inherit sensible defaults.
</section>

<section id="matchers">
  ## Matchers

  Use the `match` object to target policies by geography or jurisdiction:

  ```ts
  import { policyMatchers } from 'c15t';

  // Region-level (most specific)
  match: policyMatchers.regions([{ country: 'US', region: 'CA' }])

  // Country-level
  match: policyMatchers.countries(['DE', 'FR', 'IT'])

  // Jurisdiction-level
  match: policyMatchers.jurisdictions(['GDPR', 'UK_GDPR'])

  // Convenience helpers
  match: policyMatchers.eu()      // All EU countries
  match: policyMatchers.eea()     // EU + EEA
  match: policyMatchers.uk()      // United Kingdom
  match: policyMatchers.iab()     // IAB TCF jurisdictions

  // Combine multiple matchers
  match: policyMatchers.merge(
    policyMatchers.eu(),
    policyMatchers.uk()
  )

  // Default fallback
  match: policyMatchers.default()
  ```

  Or use the raw object form:

  ```ts
  match: { regions: [{ country: 'US', region: 'CA' }] }
  match: { countries: ['DE', 'FR'] }
  match: { jurisdictions: ['GDPR'] }
  match: { isDefault: true }
  ```
</section>

<section id="common-patterns">
  ## Common Patterns

  **The 80% case** — strict in Europe, light in California, silent everywhere else:

  ```ts
  const policies = [
    {
      id: 'policy_eu',
      match: { jurisdictions: ['GDPR'] },
      consent: { model: 'opt-in', categories: ['necessary', 'measurement', 'marketing'] },
      ui: { mode: 'banner' },
    },
    {
      id: 'policy_ca',
      match: { regions: [{ country: 'US', region: 'CA' }] },
      consent: { model: 'opt-out' },
      ui: { mode: 'banner' },
    },
    {
      id: 'policy_default',
      match: { isDefault: true },
      consent: { model: 'none' },
      ui: { mode: 'none' },
    },
  ];
  ```

  **Region overrides country** — stricter rules for California than the rest of the US:

  ```ts
  const policies = [
    {
      id: 'policy_us_ca',
      match: { regions: [{ country: 'US', region: 'CA' }] },
      consent: { model: 'opt-in', scopeMode: 'strict' },
      ui: { mode: 'banner' },
    },
    {
      id: 'policy_us',
      match: { countries: ['US'] },
      consent: { model: 'opt-out' },
      ui: { mode: 'banner' },
    },
  ];
  // US-CA → policy_us_ca (region match wins)
  // US-NY → policy_us (country match)
  ```

  **IAB in Europe, standard elsewhere:**

  ```ts
  const policies = [
    {
      id: 'policy_eu_iab',
      match: { jurisdictions: ['GDPR'] },
      consent: { model: 'iab' },
      ui: { mode: 'banner' },
    },
    {
      id: 'policy_default',
      match: { isDefault: true },
      consent: { model: 'none' },
      ui: { mode: 'none' },
    },
  ];
  // Requires iab.enabled: true in your configuration
  ```
</section>

<section id="categories-and-scope">
  ## Categories and Scope

  `consent.categories` narrows which categories c15t manages for a policy. `consent.scopeMode` controls enforcement:

  | Scope Mode | Behavior |
  |-----------|----------|
  | `'strict'` | Rejects consent submissions containing out-of-scope categories |
  | `'permissive'` | Silently filters out-of-scope categories from submissions |
  | `'*'` (wildcard) | All categories allowed |

  `consent.preselectedCategories` stages defaults in the customize dialog — useful for opt-in flows where you want certain categories pre-checked.
</section>

<section id="ui-customization">
  ## UI Customization

  Each policy can customize banner and dialog behavior independently:

  ```ts
  ui: {
    mode: 'banner',            // 'banner' | 'dialog' | 'none'
    banner: {
      allowedActions: ['accept', 'reject', 'customize'],
      primaryAction: 'accept', // Emphasized button
      actionOrder: ['reject', 'accept'], // Custom button order
      actionLayout: 'split',   // 'split' | 'inline'
      uiProfile: 'compact',   // 'balanced' | 'compact' | 'strict'
      scrollLock: true,        // Lock page scroll while banner is visible
    },
    dialog: {
      allowedActions: ['accept', 'reject', 'customize'],
      primaryAction: 'customize',
    },
  }
  ```

  <Callout type="warn">
    IAB policies (`model: 'iab'`) cannot define custom `ui.*` overrides — the IAB TCF framework controls the UI.
  </Callout>
</section>

<section id="fallbacks">
  ## Fallback Behavior

  | Configuration | Result |
  |--------------|--------|
  | `policyPacks` omitted (backend) | Legacy behavior, no runtime policy in `/init` response |
  | `policyPacks: []` (backend) | Explicit no-banner mode |
  | Non-empty pack, no match, no default (backend) | Explicit no-banner mode |
  | `offlinePolicy.policies` omitted (frontend offline) | Synthetic opt-in fallback so local dev still shows a banner |
  | `offlinePolicy: { policies: [] }` (frontend offline) | Explicit no-banner mode |

  In production, always include a default policy so unmatched traffic resolves deterministically.
</section>

<section id="snapshots-and-debugging">
  ## Runtime Outputs and Debugging

  When a backend policy pack resolves, `/init` returns:

  - `policy` — the normalized runtime policy used by the client
  - `policyDecision` — explainability metadata: `matchedBy`, stable `fingerprint`, matched geo
  - `policySnapshotToken` — an optional signed JWT that keeps later consent writes aligned with the original `/init` decision

  Useful tools while designing packs:

  - `inspectPolicies()` — surfaces validation warnings before deployment
  - `validatePolicies()` — fails fast on invalid packs (duplicate IDs, multiple defaults, IAB without `iab.enabled`)
  - DevTools Policy panel — inspects the resolved runtime decision in the browser
</section>

<section id="re-prompting">
  ## Re-Prompting on Policy Change

  When you change a policy in a way that affects consent semantics — like adding a category, changing the consent model, or modifying allowed actions — c15t automatically re-prompts returning users.

  This works through the **material policy fingerprint**: a hash of only the consent-affecting fields (model, categories, scope, allowed actions, proof settings). Presentation-only changes like copy, layout, or scroll lock do not trigger re-prompts.

  | Change | Re-prompts? |
  |--------|------------|
  | Add a consent category | Yes |
  | Change `model` from `opt-out` to `opt-in` | Yes |
  | Remove an `allowedAction` | Yes |
  | Change `uiProfile` or `actionLayout` | No |
  | Update translation copy | No |
  | Change `scrollLock` | No |
</section>

<section id="design-guidelines">
  ## Design Guidelines

  - **Start from defaults.** Use `policyPackPresets` to get running, then customize for your needs.
  - **Keep packs small.** A handful of regional policies is better than dozens of tiny fragments.
  - **Think risk, not geography.** Geography is just a matcher — the real question is what consent behavior each region needs.
  - **Always include a default.** Unless "no banner for unmatched traffic" is intentional.
  - **Keep policy IDs stable.** They appear in debugging output, snapshots, and audit records.
  - **Backend is source of truth.** Use frontend `offlinePolicy.policies` only for previews, tests, and static deployments.
  - **Use `inspectPolicies()` before deploying.** It catches overlapping matchers, missing defaults, and IAB misconfigurations.
</section>
